# CAMEL Annotation — HPC Batch Processing

Self-contained scripts for running CAMEL annotation on HPC clusters using
HTCondor and vLLM. Adapted from Prof. Introne's belief extraction HPC pipeline.

**Status:** ON HOLD — HPC cluster full, target availability early April 2026.

## Overview

| File | Purpose |
|------|---------|
| `camel_annotate_hpc.py` | Main annotation script (starts vLLM, processes chunk, saves Feather). Use `--no-server` for local testing on Apophis. |
| `generate_batch_jobs.py` | Generates per-model HTCondor submit files and wrapper scripts with GPU routing by model size. |
| `wrapper.sh` | Bash wrapper — sets up conda env and runs the script (per-model versions auto-generated by `generate_batch_jobs.py`). |
| `test_job.sub` | HTCondor submit file for test job (validates setup) |
| `split_data.py` | Splits corpus into chunks (reuse from project root) |
| `config2.yaml` | CAMEL config with prompts, labels, guidelines. Column names: `Number` (text ID), `text` (content). |

**Removed:** `merge_results.py` — merging chunk results is now done locally in `~/code_space3`, not on HPC nodes. See Step 7 below.

## Prerequisites

### 1. Singularity Container

Pull once on the login node:
```bash
singularity pull vllm-openai.sif docker://vllm/vllm-openai:v0.16.0
```

### 2. Conda Environment (HPC nodes)

Note: Apophis uses `~/myenv` (Python venv). HPC nodes use conda.
```bash
conda create -n camel-annotation python=3.11 pandas pyarrow pyyaml requests
conda activate camel-annotation
```

### 3. Pre-download Model Weights

To avoid download failures during jobs:
```bash
huggingface-cli download casperhansen/llama-3.3-70b-instruct-awq
huggingface-cli download Qwen/Qwen2.5-72B-Instruct-AWQ
huggingface-cli download Valdemardi/DeepSeek-R1-Distill-Qwen-32B-AWQ
```

### 4. Create Directories

Directories are now symlinks to `/DATA`. If setting up fresh:
```bash
mkdir -p /DATA/szkhan/camel/hpc_data/chunks
mkdir -p /DATA/szkhan/camel/hpc_data/results
mkdir -p /DATA/szkhan/camel/hpc_data/logs
ln -s /DATA/szkhan/camel/hpc_data/chunks chunks
ln -s /DATA/szkhan/camel/hpc_data/results results
ln -s /DATA/szkhan/camel/hpc_data/logs logs
```

### 5. Copy Config
```bash
cp ~/code_space3/config2.yaml .
```

## Quick Start

### Step 1: Prepare Corpus as Feather

If your corpus is CSV:
```python
import pandas as pd
df = pd.read_csv("/DATA/szkhan/camel/samples/CAMEL_cleaned_data_complete.csv")
df.to_feather("/DATA/szkhan/camel/samples/CAMEL_cleaned_data_complete.feather")
```

### Step 2: Create Test Chunk
```bash
python split_data.py --input /DATA/szkhan/camel/samples/CAMEL_cleaned_data_complete.feather --create-test --test-size 100
```

### Step 3: Run Test Job
```bash
condor_submit test_job.sub
```

Monitor:
```bash
condor_q $USER
tail -f logs/test_job.out
```

### Step 4: Verify Test Results
```python
import pandas as pd
df = pd.read_feather("results/test_results.feather")
print(f"Rows: {len(df)}, Columns: {len(df.columns)}")
print(df[["text_id", "prompt_name", "model", "Care", "Honesty"]].head())
```

### Step 5: Split Production Data
```bash
python split_data.py --input /DATA/szkhan/camel/samples/CAMEL_cleaned_data_complete.feather --chunk-size 5000
```

This creates:
- `chunks/chunk_000.feather`, `chunk_001.feather`, ...

### Step 6: Generate Per-Model Submit Files and Run Production Batch
```bash
python generate_batch_jobs.py
condor_submit llama33_70b_submit.sub
condor_submit qwen25_72b_submit.sub
condor_submit deepseek_r1_32b_submit.sub
```

Monitor:
```bash
condor_q $USER
watch -n 60 'ls -la results/*.feather | wc -l'
```

### Step 7: Merge Results (Locally, NOT on HPC)

After all jobs complete, merge in `~/code_space3` (not in this directory):
```bash
cd ~/code_space3
python -c "
import pandas as pd
from glob import glob
files = sorted(glob('hpc/results/*.feather'))
merged = pd.concat([pd.read_feather(f) for f in files])
merged.to_feather('outputs/full_results.feather')
print(f'Merged {len(files)} chunks, {len(merged)} rows')
"
```

### Step 8: Evaluate
```bash
cd ~/code_space3
python evaluate.py --results outputs/full_results.feather --gold samples/CAMEL_cleaned_data_complete_ans.feather
```

## Running Multiple Models

Each model gets a separate batch via `generate_batch_jobs.py`. You can also
set model via environment variables in `wrapper.sh` or submit separate jobs:
```bash
# Llama 3.3 70B (default)
condor_submit llama33_70b_submit.sub

# Qwen 2.5 72B
condor_submit qwen25_72b_submit.sub

# DeepSeek R1 32B (auto-detects higher max_tokens for reasoning chains)
condor_submit deepseek_r1_32b_submit.sub

```

## GPU Routing by Model Size

| Model | Size | GPUs | TP | Node Pool |
|-------|------|------|----|-----------|
| Llama 3.3 70B | 70B | 2 | 2 | 2-GPU high-VRAM only |
| Qwen 2.5 72B | 72B | 2 | 2 | 2-GPU high-VRAM only |
| AceGPT v2 70B | 70B | 2 | 2 | 2-GPU high-VRAM only |
| DeepSeek R1 32B | 32B | 1 | 1 | Any node with 1 GPU |
| Mistral Small 24B | 24B | 1 | 1 | Any node with 1 GPU |
| Falcon-H1 34B | 34B | 1 | 1 | Any node with 1 GPU |

## Resource Requirements

| Model Size | GPUs | GPU VRAM | CUDA | CPU | RAM |
|-----------|------|----------|------|-----|-----|
| 70B (Llama, Qwen, AceGPT) | 2 | ≥40GB each | ≥12.8 | 8 | 32GB |
| 32-34B (DeepSeek, Falcon-H1) | 1 | ≥40GB | ≥12.8 | 8 | 32GB |
| 24B (Mistral Small) | 1 | ≥24GB | ≥12.8 | 8 | 32GB |

## Output Format

Results are Feather files in wide format (one row per text × prompt):

| Column | Description |
|--------|-------------|
| `text_id` | Original text identifier |
| `prompt_id` | Prompt strategy ID |
| `prompt_name` | Prompt strategy name |
| `model` | Model identifier |
| `original_index` | Index in original corpus (for merging) |
| `Care`, `Honesty`, ... | 25 label columns (1=YES, 0=NO, -1=error) |
| `response__Care`, ... | 25 response text columns |

To test without Singularity (uses the local vLLM server on port 8900):

## Troubleshooting

### vLLM server doesn't start
```bash
cat logs/<job_name>.err
cat vllm_server.log  # in working dir
```

### Jobs stuck in queue
```bash
condor_status -constraint 'TotalGpus >= 2' -af Machine CUDADriverVersion CUDAGlobalMemoryMb
```

### Check available GPU nodes
```bash
condor_status -constraint 'TotalGpus >= 1' -af Machine TotalGpus CUDAGlobalMemoryMb
```
